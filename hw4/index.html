Overview

In this homework, we built a real-time cloth simulation from a grid of point masses and springs. The cloth starts as a flat set of particles. Springs connect nearby particles, gravity pulls the particles downward, and each simulation step updates the particles using Verlet integration. After the particles move, the simulator corrects cases where springs stretch too far, handles collisions with objects like spheres and planes, and uses spatial hashing so the cloth can collide with itself without checking every pair of particles.

The main idea we learned is that cloth behavior comes from many small local rules working together. No single particle knows what the whole cloth should look like. Each particle only responds to forces, constraints, and collisions near it. When all of those local updates happen every frame, the result looks like a flexible sheet.

We also implemented several GLSL shaders. These shaders move lighting and material calculations onto the GPU. The vertex shader prepares per-vertex data such as position, normal, tangent, and texture coordinate. The fragment shader then uses interpolated versions of those values to decide the color of each visible point on the cloth or sphere.

Part 1: Masses and Springs

We represented the cloth as a regular grid of point masses. For a horizontal cloth, the grid lies in the xz-plane with a fixed y value of 1. For a vertical cloth, the grid lies in the xy-plane, with a very small random z offset so that the cloth does not start as a perfectly flat vertical sheet. The point masses are stored in row-major order, which means all points in the first row come first, then all points in the second row, and so on.

After creating the masses, we added three kinds of springs. Structural springs connect each mass to the mass directly left and directly above it. Shearing springs connect diagonal neighbors. Bending springs connect masses that are two steps apart horizontally or vertically. Structural springs preserve the basic grid, shearing springs resist diagonal collapse, and bending springs make the cloth less likely to fold sharply at every grid edge.

Part 2: Simulation via Numerical Integration

Each simulation step starts by clearing the force stored on every point mass. We then add the external force from gravity using \(F = ma\). After that, we loop through the springs and apply Hooke's law. If a spring is longer than its rest length, it pulls its endpoints toward each other. If it is shorter, it pushes them apart. We used a smaller spring constant for bending springs so that bending is softer than stretching or shearing.

To update positions, we used Verlet integration. Instead of storing velocity directly, Verlet estimates velocity from the difference between the current position and the previous position. The update is \[ x_{t+\Delta t} = x_t + (1 - d)(x_t - x_{t-\Delta t}) + a_t\Delta t^2. \] Pinned points skip this update, so they stay fixed in place.

At the end of each step, we enforce the Provot constraint. If any spring becomes more than 10 percent longer than its rest length, we move the endpoints closer together. If both endpoints can move, each gets half of the correction. If one endpoint is pinned, the unpinned endpoint gets the full correction.

Parameter Effects

Increasing ks makes the springs stronger. With a low ks, the cloth stretches more and looks looser before it settles. With a high ks, the cloth resists stretching more strongly, so it keeps its shape better and can look stiffer.

Increasing density makes every point mass heavier. Gravity produces a larger force, but acceleration from the same spring force becomes smaller because the mass is larger. In practice, denser cloth tends to sag more and respond more slowly to spring corrections.

Increasing damping removes more motion from the Verlet update. Low damping lets the cloth swing and bounce for longer. High damping makes it settle faster, but if damping is too strong the cloth can look unnaturally sluggish.

Part 3: Handling Collisions with Other Objects

For sphere collisions, we check whether a point mass is inside or on the sphere. If it is, we find the point on the sphere surface in the same direction from the center. That surface point is the target position the point mass should have reached instead of going into the sphere. We then apply the correction from the previous position toward that target, scaled by \(1 - friction\), so friction reduces how far the point slides.

For plane collisions, we compare which side of the plane the point mass was on last frame and which side it is on now. If the signs differ, the point crossed the plane. We compute where the straight path between the two positions intersects the plane, then place the point slightly above the plane on the side it came from. We again scale the correction by \(1 - friction\).

Part 4: Handling Self-Collisions

A direct self-collision check would compare every point mass against every other point mass, which is too slow for real-time simulation. Instead, we used spatial hashing. At the start of each step, we divide space into boxes and store each point mass in a hash table based on the box that contains it. Then each point mass only checks against other masses in the same box.

During self-collision, we check whether another point mass is closer than twice the cloth thickness. If two masses are too close, we compute a correction that would separate them to exactly that distance. A point may have several nearby neighbors, so we average all of the correction vectors and divide by the number of simulation steps. This keeps the correction from being too sudden.

Higher density makes the cloth feel heavier, so the folds tend to press downward more. Lower density gives the spring forces more influence relative to the mass, so the cloth can react more sharply. Lower ks makes the cloth stretch and fold more loosely, while higher ks makes it resist stretching and keep a crisper sheet-like shape.

Part 5: Shaders

What a Shader Program Does

A shader program is a pair of small programs that run on the GPU. The vertex shader runs once per vertex. It transforms the vertex position into screen space and passes useful values, such as world position, normal, tangent, and texture coordinate, to the next stage. The fragment shader runs for each visible fragment after the triangle is rasterized. It uses the interpolated values from the vertex shader to compute the final color.

Diffuse and Blinn-Phong Shading

Diffuse shading models a matte surface. A surface is brightest when it faces the light directly and darkest when it faces away. We compute this with the dot product between the surface normal and the direction to the light.

Blinn-Phong shading adds two more pieces. The ambient term gives the object a small base amount of light, even if it is not facing the light. The diffuse term handles matte lighting. The specular term creates shiny highlights by comparing the surface normal to the halfway direction between the light direction and view direction. The full result is the sum of ambient, diffuse, and specular lighting.

Texture Mapping

For texture mapping, we used the interpolated UV coordinate to sample a 2D texture. The sampled texture color becomes the output color of the fragment. This lets the same cloth mesh display detailed color variation without adding more geometry.

Bump and Displacement Mapping

Bump mapping changes the normal used for lighting, but it does not move the actual vertices. We sampled nearby values in the height map to estimate how the surface height changes in the \(u\) and \(v\) directions. Those changes define a new local normal, which we transform into world space using the tangent, bitangent, and normal basis.

Displacement mapping uses the same idea for normals, but it also moves each vertex along its normal direction based on the height map. Because the geometry itself changes, displacement mapping shows real silhouette and shape changes. It depends more heavily on mesh resolution. With a coarse sphere, the displaced shape has fewer geometric details. With a fine sphere, the displacement can show much smoother detail.

Mirror Shader

The mirror shader reflects the view direction around the surface normal, then uses that reflected direction to sample a cubemap. This approximates a reflective material by looking up what the environment would look like in the reflected direction.

Custom Shader

We made a custom shader that combines texture color, a procedural stripe pattern, Blinn-Phong lighting, and a small amount of cubemap reflection. The goal was to make a material that still responds to light like cloth, but has more visual variation than a plain color.
